# GitHub App Identities

uio supports dedicated GitHub App identities for agents that act on GitHub — separate
non-human service accounts for planning, coding, and reviewing work. Each identity has a
distinct permission scope, enforcing separation of duties at the GitHub App layer.

## The three identities

| Identity | Frontmatter value | Operations |
|---|---|---|
| AI Planner | `planner` | Create issues · Comment on issues and PRs · Summarize milestones |
| AI Coder | `coder` | Create branches · Commit code · Open pull requests |
| AI Reviewer | `reviewer` | Read diffs · Post structured review comments |

## Declaring an identity

Add `vcs-identity` to your agent's frontmatter:

```yaml
---
name: my-github-agent
description: An agent that opens pull requests.
vcs-identity: coder   # planner | coder | reviewer
capabilities:
  - vcs
---
```

> **Deprecated alias:** `github-identity` is accepted for backwards compatibility but
> new definitions should use `vcs-identity`.

uio obtains a short-lived GitHub App installation token before the agent loop starts and
sets `GH_TOKEN` — both the `gh` CLI and GitHub MCP server pick it up automatically.

## Attribution system

When `vcs-identity` is set in an agent's frontmatter, the uio runner automatically
injects an attribution block into the system prompt before the agent body.
The block is produced by `build_attribution_instructions()` in
`uio/core/attribution.py` and injected immediately before the agent body for
every agent run that carries a recognised `vcs-identity` role (`planner`,
`coder`, or `reviewer`).

### What the injected block contains

The attribution block instructs the LLM to append identity footers to every
piece of VCS content it generates:

| Content type | What is appended |
|---|---|
| Issue comments and PR/MR comments | A one-line footer: `🤖 Generated by **uio / <Display Name>** · <agent-name> · uio <version>` |
| Pull request / merge request bodies | A multi-line disclosure block identifying the PR as AI-generated |
| Git commits (coder only) | Instructions to set `user.name` / `user.email` to the bot identity on every `git commit` |

The footer values are rendered at runtime from the agent's `name` frontmatter
field, the resolved role, and the running uio version.

### Example

Given this agent definition:

```yaml
---
name: my-coder-agent
description: Opens pull requests for approved tasks.
vcs-identity: coder
capabilities:
  - vcs
---
```

The runner injects an attribution block that:

1. Identifies the agent as **uio AI Coder** acting as a non-human GitHub App identity.
2. Requires every issue or PR comment body to end with:
   ```
   ---
   🤖 Generated by **uio / AI Coder** · my-coder-agent · uio 1.2.3
   ```
3. Requires every PR description to end with a disclosure block:
   ```
   ---
   > **AI-generated pull request** — authored by the uio AI Coder agent.
   > Human review is required before merging.
   > Powered by [uio](https://github.com/jomkz/uio).
   ```
4. Requires every `git commit` to use the bot author identity:
   ```
   git -c user.name="uio AI Coder" -c user.email="uio-coder[bot]@users.noreply.github.com" commit …
   ```

Agent authors do not need to handle attribution manually — uio injects these
rules automatically whenever `vcs-identity` is declared.

### GitLab provider

When `vcs-provider: gitlab` is also set in the frontmatter, the injected block
uses GitLab-specific terminology (merge request instead of pull request) and the
bot email domain becomes `users.noreply.gitlab.com`.
