# AI Governance — Attribution Standard

> Part of epic #53 (Enterprise GitHub Identity Architecture for AI Agents).
> Covers the attribution rules for all AI-generated GitHub activity produced by uio agents.

---

## Why attribution matters

uio agents that act on GitHub (creating issues, opening PRs, committing code, posting
reviews) use dedicated GitHub App identities rather than personal accounts. Even with
separate App identities, the GitHub UI does not always make it immediately obvious that
a comment or PR was AI-generated. The attribution standard ensures:

- Human reviewers can immediately identify AI-authored content.
- AI-generated activity is auditable and filterable in GitHub search.
- The project satisfies disclosure requirements before any production rollout.

---

## Attribution templates

The templates below are the canonical standard. They are injected automatically by the
uio runner into the system prompt whenever `github-identity` is set — agent authors do
not need to add them manually.

### Issue and PR comment footer (all identities)

Appended to the body of every `gh issue comment`, `gh pr comment`, or MCP equivalent:

```
---
🤖 Generated by **uio / AI Planner** · github-planner · uio 0.1.0
```

The identity name (`AI Planner`, `AI Coder`, `AI Reviewer`), agent name, and uio
version are filled in at runtime.

### Pull request description footer (AI Coder)

Appended to the body of every PR opened by the AI Coder:

```
---
> **AI-generated pull request** — authored by the uio AI Coder agent.
> Human review is required before merging.
> Powered by [uio](https://github.com/jomkz/uio).
```

### Commit author (AI Coder)

All commits created by the AI Coder must use the non-human author identity:

| Field | Value |
|---|---|
| `user.name` | `uio AI Coder` |
| `user.email` | `uio-coder[bot]@users.noreply.github.com` |

The runner injects instructions into the system prompt requiring the LLM to pass
`-c user.name=… -c user.email=…` flags on every `git commit` call.

Non-human author identities for reference:

| Identity | `user.name` | `user.email` |
|---|---|---|
| AI Planner | `uio AI Planner` | `uio-planner[bot]@users.noreply.github.com` |
| AI Coder | `uio AI Coder` | `uio-coder[bot]@users.noreply.github.com` |
| AI Reviewer | `uio AI Reviewer` | `uio-reviewer[bot]@users.noreply.github.com` |

## How injection works

When `github-identity` is set in an agent's frontmatter, `uio/core/runner.py` calls
`build_attribution_instructions()` from `uio/core/attribution.py` and prepends the
resulting instructions block to the system prompt, between the runtime preamble and
the agent body. The agent author writes their workflow instructions as normal; the
attribution rules are layered on top automatically.

The `uio/core/attribution.py` module also exposes individual rendering functions for
use in tests and tooling:

| Function | Returns |
|---|---|
| `render_comment_footer(role, agent_name, version)` | Footer string for comments |
| `render_pr_footer(role)` | Disclosure block for PR bodies |
| `render_commit_author(role)` | `{"name": …, "email": …}` dict |
| `build_attribution_instructions(role, agent_name, version)` | Full system-prompt block |

---

## Governance rules

1. **Mandatory for all GitHub App identities.** Any agent with `github-identity` set
   must include attribution footers on all GitHub content. The runner enforces this
   via system prompt injection — it is not optional.

2. **Humans must approve AI-generated code.** Branch protection rules on `main` and
   `develop` require at least one human approving review before an AI Coder PR can be
   merged. AI Reviewer approval alone does not satisfy this requirement.

3. **No merge authority.** None of the three App identities has permission to merge
   pull requests. Merge is always a human action.

4. **Quarterly review.** App permissions and governance rules are reviewed quarterly
   against the principle of least privilege. Any permission no longer required is
   revoked.

5. **Escalation path.** Adding a capability currently excluded from an identity
   requires opening a `ai-governance`-labelled issue, human approval, and a
   minimum two-week trial in a non-production repository before wider rollout.
